Requisitos Funcionais, Não Funcionais, Inversos e não Técnicos

 

1. Introdução

 

Os requisitos de um sistema são as declarações textuais das atividades que o sistema fará (Funcionais) ou o que não permitirá que se faça (Inversas), uma vez concluído seu desenvolvimento. Visto o alto nível de redundância nas diversas interfaces do sistema quanto aos requisitos funcionais. Vamos apontar a seguir os requisitos comuns a todos os módulos, e a seguir, apenas os requisitos individuais. Desta forma será mais fácil e rápido o apontamento dos requisitos do sistema como um todo e o de um dado módulo do mesmo.

 

o        Requisitos Funcionais

o        O sistema deve fazer a crítica e login dos usuários, permitindo o acesso somente as partes do sistema liberadas a este usuário;

o        O sistema deve prever o acesso a usuários visitantes, limitando o escopo de atuação deste tipo de usuário a apenas consultas aos dados armazenados;

o        O sistema deve obedecer as regras de integridade definidas no banco de dados;

o        O sistema deve criar um identificador para cada informação armazenada;

o        O sistema deve exigir o preenchimento dos campos obrigatórios para o armazenamento da informação;

o        Deve existir mecanismo de pesquisa para as informações armazenadas no banco de dados;

o        O sistema deve operar na área definida na interface como a área de interação com o usuário, não fazendo scroll da tela, exceto nas exceções apontadas;

o        O sistema deve manipular as janelas de pop-up de forma a fecha-las após determinado tempo;

o        O sistema deve emitir um Aviso de Tombamento para todos os usuários vinculados à aquela coleção;

o        O sistema deve emitir um Aviso de Alteração na Ficha Técnica quando os técnicos de restauração fazem modificações em informações pertinentes à catalogação (técnica e dimensões da obra) na ficha de restauro;

o        O sistema deve prover um mecanismo de geração de Agenda e Notas para ser utilizado como comunicação interna entre usuários e também como agenda, onde o usuário insere uma determinada nota (lembrete) para ele mesmo numa data futura;

o        O usuário deverá poder marcar/desmarcar uma nota como lida e seleciona a exibição de lidas/não lidas;

o        Deve existir um parâmetro para determinar o tempo de armazenamento da agenda tomando como base a data de agendamento e não de criação da nota de forma a evitar a exclusão de notas criadas com antecedência;

 

o        Requisitos não Funcionais

o        O sistema exige o servidor HTTP Apache, não sendo possível a utilização de outro servidor;

o        O sistema será escrito em PHP;

o        O sistema armazenará seus dados em um banco de dados MySQL;

o        O sistema será executado a partir das estações clientes em um browser (Internet Explorer ou FireFox);

o        O sistema rodará em um servidor dedicado a ele;

o        O sistema exige infraestrutura de rede (TCP/IP) para seu funcionamento;

 

 

o        Requisitos Inversos

o        O sistema não deverá permitir o acesso a usuários não autorizados, nem o acesso a rotinas não disponibilizadas previamente a um usuário através do administrador do sistema;

o        O sistema não deve desprezar informações que foram entradas nas telas da interface;

o        O sistema não deve aceitar entrada de dados não compatíveis com o formato do campo no banco de dados;

o        O sistema não deve permitir o cadastramento de datas inválidas, mesmo considerando-se imprecisões na localização temporal do acontecimento;

o        O sistema não deve abrir duas janelas de pop-up com o mesmo conteúdo;

o        O sistema não deve permitir o cadastramento parcial de informações;

 

 

o        Requisitos não Técnicos

o        Os fontes do sistema serão de propriedade do cliente;

o        O sistema deverá ser entregue no prazo acordado contratualmente ou posteriormente adendado;

o        O sistema deverá ter seu custo dentro dos valores acertados contratualmente ou posteriormente adendado;

o        O sistema será desenvolvido nas instalações da contratada;

o        O sistema deverá ser disponibilizado para acompanhamento da contratante a partir de link da contratada enquanto estiver na fase de desenvolvimento;

 

Tendo sido visto os requisitos comuns ao sistema, a partir de agora veremos os requisitos dos módulos do sistema. Vale lembrar que ainda assim alguns requisitos são repetitivos. Além disso, os requisitos não Funcionais podem não ser representativos, e os requisitos não Técnicos não se aplicarem.

 

Acervo

·         Obras

o        Requisitos Funcionais

§         O sistema deve permitir a inclusão de fichas de obras, obedecendo os critérios definidos para a criação de uma nova ficha de obras e gerando log desta ação;

§         O sistema deve abrir a ficha da obra ao se clicar na linha referente a esta obra na relação devolvida pela consulta;

§         O sistema deve permitir a manutenção de uma ficha de obra, gerando o log destas ações;

§         O número de registro (tombo) é gerado manualmente a partir do livro de tombo.

§         O sistema deverá dividir a ficha em Obra e Partes da Obra;

§         O sistema deverá fazer o controle de acesso por coleção;

§         O sistema deverá registrar as alterações na ficha da obra em log, contendo o usuário e data/hora de alteração.

§         O sistema deverá implementar o relacionamento entre  obras;

§         O acesso ao manual se faz clicando no label do campo.

o        Requisitos não Funcionais

§         O sistema deve nesta opção trabalhar na área da interface da barra de identificação até o fim;

§         O sistema deve organizar a entrada de dados da maneira mais lógica possível, facilitando o trabalho de cadastramento da obra;

o        Requisitos Inversos

§         O sistema não deve permitir o cadastramento da mesma obra mais de uma vez;

§         O sistema não deve permitir a criação da obra sem a vinculação ao autor ou autores;

 

 

·         Imagens

o        Requisitos Funcionais

§         O sistema deve ter função para manutenção do banco de imagens do sistema;

§         O sistema deve permite localizar a ficha de imagem através de critérios básicos de seleção, ou a abertura de uma nova ficha;

§         O sistema deve ao clicar em uma determinado nome de imagem, o sistema abre a respectiva ficha para edição;

§         O sistema deve permitir a impressão de laudos;

§         O sistema deve possibilitar a seleção da imagem que será utilizada no laudo;

§         O sistema deve prover o campo “Forma de exibição”, com duas opções 1-Altura dos olhos (default) e 2-No chão. O sistema deve utilizar este campo para estabelecer o posicionamento da imagem da Obra na função Escala, mais especificamente para pequenas esculturas que seriam normalmente exibidas ao nível dos olhos, mas que devam ser exibidas no chão;

§         O sistema deve possibilitar ao catalogador fazer o upload e preenchimento da ficha de imagem;

§         A função Escala deve seguir os critérios atuais de funcionamento;

o        Requisitos Inversos

§         O sistema não deve permitir o cadastramento de uma imagem sem a devida vinculação ao arquivo de imagem;

 

·         Movimentação

o        Requisitos Funcionais

§         O sistema deve ter função para registro de movimentação de obras do acervo;

§         O sistema deve permitir o cadastro de exposições, no caso de movimentação externa;

§         O sistema deve permitir o registro de movimentações internas entre os locais cadastrados no sistema;

§         A emissão de laudos é diferenciada para o caso de exposições itinerantes, onde uma movimentação abrange diversos locais. Deve haver um laudo para cada local de destino;

§         O sistema deve permitir o registro da movimentação de um conjunto de obras em uma só operação;

§         O sistema deve fazer a diferenciação de data de saída e possível retorno e das datas de exposição;

 

o        Requisitos Inversos

§         O sistema deve ao fazer movimentação entre setores de movimentação controlada, solicitar um aceite no recebimento da obra pelo setor de destino;

§         O sistema não deve permitir a movimentação de uma mesma obra, no mesmo tempo, para mais de um lugar;

 

·         Relacionamento

o        Requisitos Funcionais

§         O sistema deve permitir registrar os diversos relacionamentos entre as obras do acervo, tais como cópia, esboço, original de, série, catálogo;

§         Esta funcionalidade (relacionamento) deve permitir consultar obras relacionadas a partir da ficha de uma obra;

§         O relacionamento deve permitir nominar o sentido do relacionamento;

§         No sistema os relacionamentos do tipo Album/caderno/série, devem permitir a pesquisa pelo nome do conjunto e recuperar as obras que compõem este conjunto;

o        Requisitos não Funcionais

§         Deve-se fazer o levantamento de todos os tipos de relacionamento possíveis para a inclusão na tabela de Relacionamentos;

o        Requisitos Inversos

§         O sistema não deve permitir o cadastramento de relacionamentos que não estejam previstos na tabela de relacionamento;

 

·         Exclusão

o        Requisitos Funcionais

§         O sistema deve permitir a exclusão da ficha técnica de uma obra e seus registros dependentes;

§         O sistema somente deve permitir o acesso a esta rotina ao administrador do sistema;

o        Requisitos Inversos

§         O sistema não deve permitir a exclusão parcial de uma ficha técnica;

 

Autores

·         Cadastro

o        Requisitos Funcionais

§         O sistema deve ter função para manutenção do cadastro de autores de obras registradas no sistema;

§         O sistema deve permitir consultar/editar a bibliografia e exposições relacionadas ao autor;

§         O sistema deve permitir o acesso às obras relacionadas ao autor;

 

·         Exclusão

o        Requisitos Funcionais

§         O sistema deve excluir o cadastro de um determinado autor selecionado;

§         Esta função deve ter o acesso restrito ao administrador do sistema

§         Apenas autores sem obras relacionadas podem ser excluídos do sistema

o        Requisitos Inversos

§         Não se deve permitir a exclusão de autores com obras cadastradas;

 

 

Restauração

§         Requisitos Funcionais

 

§         O sistema deve manter o modelo de tela atual, entretanto o campo “Tratamento” passa a ser único para a ficha e não por extrato;

§         O sistema deverá prover uma simplificação na inserção de fotos das etapas de restauro;

§         O Nº de registro no restauro passa a ser o próprio nº de registro da Obra (tombo) + seqüencial de entrada no restauro;

§         O campo I.R. (Inventário de Restauro) deve ser mantido para fins de compatibilidade com a versão atual. Deve haver seleção da ficha por este campo;

§         O sistema deve permitir a entrada da obra na restauração sem a necessidade de abrir uma ficha de restauração;

§         O preenchimento do campo Tratamento (texto) deve ter a funcionalidade existente na versão atual do módulo de Pintura, com seleção de termos chave para evitar-se erros de grafia;

§         O sistema deve, quando os técnicos de restauração fizerem modificações em informações pertinentes à catalogação (técnica e dimensões da obra) na ficha de restauro, avisar ao catalogador da obra em forma de sugestão de alteração;


Consultas

Atenção: Todas as consultas tem o mesmo Requisito Inverso, a saber:

 

q       O sistema não deve devolver informações inexistentes;

q       O sistema não deve devolver a informação em duplicidade;

q       O sistema não deve deixar de responder a uma consulta, mesmo que tenha nenhum registro como resposta;

q       O sistema deve validar inicialmente os parâmetros da consulta, evitando consultas inconsistentes;

q       O sistema deve intervir quando a consulta entrar em situações de looping;

 

Abaixo as consultas e seus Requisitos Funcionais:

 

·         Obras

o        Pesquisa

§         Requisitos Funcionais

·         O sistema deve permitir efetuar pesquisas no acervo do museu;

·         O sistema deve permitir a combinação de diversos critérios de seleção, a saber: título, autor (nome, país/estado de origem, estado de atuação), coleção, técnica, período, temas, exposições, escola, estilos, local de produção, dimensões e campos descritivos;

·         O sistema, nesta função deverá ter a opção de exibir ou não miniaturas da imagem principal da obra;

 

o        Obras mais consultadas

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com as “n” (parâmetro) obras mais consultadas na função “Pesquisa”. O quadro irá mostrar o título da obra, a quantidade total de consultas nos últimos “n” dias (parâmetro) e o valor percentual em relação ao total de consultas;

·         No sistema deverá existir link no título da obra para acesso à ficha técnica de consulta da obra;

 

o                   Coleçoes mais consultadas

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com as “n” (parâmetro) coleçoes mais consultadas na função “Pesquisa”. O quadro irá mostrar o título da obra, a quantidade total de consultas nos últimos “n” dias (parâmetro) e o valor percentual em relação ao total de consultas;

·         Autores

o        Pesquisa

§         Requisitos Funcionais

·         O sistema deve permitir efetuar pesquisas no cadastro de autores do museu, combinando diversos critérios de seleção, a saber: nome, local de nascimento, local de falecimento, estado de atuação, país de atuação, bibliografia e biografia (texto);

 

·         Autores mais consultados

o        Pesquisa

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com os “n” (parâmetro) autores mais consultados na função “Pesquisa”. O quadro irá mostrar o nome do autor, a quantidade total de consultas nos últimos “n” dias (parâmetro) e o valor percentual em relação ao total de consultas;

·         No sistema deverá existir link no nome do autor para acesso à consulta ao cadastro do autor;

 

·         Autores por Coleção

o        Pesquisa

§         Requisitos Funcionais

·         O sistema deverá exibir lista de autores relacionados a uma determinada coleção selecionada;

·         No sistema deverá existir link no nome do autor para acesso à consulta ao cadastro do autor;

 

·         Estatística

o        Autores - % por estado de nascimento

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com a quantidade e percentual do número de autores por estado de nascimento no cadastro;

·         No sistema deverá existir link no nome do estado para acesso à relação de autores do respectivo estado;

·         No sistema deverá existir link no nome do autor para acesso à consulta ao cadastro do autor;

 

o        Autores - % por país de nascimento

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com a quantidade e percentual do número de autores por país de nascimento no cadastro;

·         No sistema deverá existir link no nome do país para acesso à relação de autores do respectivo estado;

·         No sistema deverá existir link no nome do autor para acesso à consulta ao cadastro do autor;

 

o        Acervo - % por autor

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com a quantidade e percentual do número de obras por autor;

·         Serão considerados os “n” (parâmetro) primeiros autores com maior número de obras

 

o        Acervo - % por coleção

§         Requisitos Funcionais

·         O sistema deverá ter um quadro resumo com a quantidade e percentual do número de obras por coleção;

·         Serão considerados as “n” (parâmetro) primeiras coleções com maior número de obras

Agenda

·         Anotações

o        Requisitos Funcionais

§         O sistema deve permitir visualizar e criar anotações na agenda pessoal do usuário do sistema;

§         O sistema deve permitir que o usuário determine uma data/hora para lembrete e o sistema deve gerar o alerta futuramente;

 

·         Notas do dia

o        Requisitos Funcionais

§         O sistema deve exibir as anotações inseridas no dia atual e também as anotações com data de lembrete para o dia atual;

 

·         Mensagens

o        Requisitos Funcionais

§         O sistema deve exibir relação das mensagens recebidas e ao clicar-se sobre uma mensagem o texto da mesma é exibido;

§         O sistema deve permitir o envio de mensagens p/ qualquer usuário do sistema;

§         O sistema deve indicar na tela principal, se existe mensagem recebida ainda não lida pelo usuário;

Estrutura

·         Instituição

o        Requisitos Funcionais

§         O sistema deverá prover o cadastro de informações da instituição onde o sistema Donato está sendo executado;

§         Esta informação deve ser preenchida quando da instalação do sistema;

o        Requisitos Inversos

§         O sistema não permitirá mais de uma entrada no cadastro de Instituição;

 

·         Tabelas

o        Requisitos Funcionais

§         O sistema deverá permitir a inclusão, alteração e exclusão de dados auxiliares utilizados em diversas funções do sistema;

o        Requisitos Inversos

§         Uma vez existindo uma tabela auxiliar, o sistema não deve permitir o cadastramento de informação que não esteja validada na devida tabela;

§         As tabelas auxiliares não devem permitir cadastramento em duplicidade;

 

·         Manual de catalogação

o        Requisitos Funcionais

§         O sistema deverá permitir a manutenção do manual de catalogação on-line, disponibilizado nas telas de cadastro do sistema;

·         Sistema

o        Usuários

§         Requisitos Funcionais

·         O sistema deverá permitir o cadastramento de usuários com acesso ao sistema Donato;

·         O sistema deverá fazer o controle do acesso aos itens de menu do sistema;

·         O sistema deverá fazer o controle do acesso às coleções;

·         Esta função deve ser de uso exclusivo do(s) administrador(es) do sistema;

 

o        Alterar senha

§         Requisitos Funcionais

·         Esta função deve ser disponibilizada para que cada usuário do sistema possa alterar a própria senha periodicamente;

§         Requisito não Técnico

·         Deverá existir um login para cada usuário, não sendo permitido que dois ou mais usuários façam uso de um mesmo login;

§         Requisitos Inversos

·         Dois usuários não podem ter a mesma identificação;

·         Uma vez que o login de um usuário esteja desabilitado, este não poderá fazer acesso ao sistema;

 

o        Menus

§         Requisitos Funcionais

·         O sistema deverá permitir o cadastramento da árvore de menus do sistema

·         Esta função é de uso exclusivo dos desenvolvedores do sistema;

 

o        Parâmetros

§         Requisitos Funcionais

·         O sistema deve permitir configurar os parâmetros que determinam o comportamento de determinadas funções do sistema, de acordo com a necessidade de cada instituição;

·         Atenção: alguns parâmetros são de uso exclusivo dos desenvolvedores do sistema;

 

o        Auditoria

§         Requisitos Funcionais

·         O sistema deve permitir a consulta ao log de alterações e exclusões das fichas técnicas de obras;

·         O sistema deve permitir a identificação dos usuários responsáveis pelas operações realizadas no sistema;

·         O sistema deve permitir a identificação da data e hora das operações realizadas no sistema;

§         Requisitos Inversos

·         As manipulações de ficha técnica e obra não podem deixar de gerar os respectivos logs;
